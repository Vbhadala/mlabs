## Issues

### Issue 1: `next.config.ts` ESM/CJS conflict — build + dev both broken

- **Severity:** **critical**
- **Repro:**
  1. `rm -rf apps/web/.next` (clean state)
  2. `pnpm --filter @mlabs/web build`
  3. Build aborts immediately at config load.
- **Expected:** Build proceeds; standalone output materializes at
  `apps/web/.next/standalone/apps/web/server.js`.
- **Actual:**
  ```
  ⨯ Failed to load next.config.ts
  ReferenceError: exports is not defined in ES module scope
  This file is being treated as an ES module because it has a '.js' file extension
  and 'apps/web/package.json' contains "type": "module".
      at <unknown> (next.config.compiled.js:2:23)
  ```
- **Console errors:** As above.
- **Suspected cause:** Next.js 16 compiles `next.config.ts` to
  `.next/types/next.config.compiled.js` with **CommonJS** semantics
  (`exports = …`), but `apps/web/package.json` declares `"type": "module"`,
  which makes Node treat the `.js` output as ESM. T9
  (`apps/web/next.config.ts`) added `import.meta.url` + `__dirname` rebind to
  configure `outputFileTracingRoot`, which seems to push Next's TS-config
  compiler into the CommonJS code path. BetFrnd's working version is
  `apps/web/next.config.mjs` (not `.ts`) — `.mjs` is unambiguously ESM and
  bypasses the conflict entirely.
- **Fix plan:**
  Rename `apps/web/next.config.ts` → `apps/web/next.config.mjs`. Drop the
  `import type { NextConfig }` line; preserve typing via `/** @type {import('next').NextConfig} */`
  JSDoc comment. Matches BetFrnd's shipped pattern (verified
  `apps/web/next.config.mjs` in betfrnd-readonly clone). Also blocks S3.
- **Blocks:** S2, S3, the deploy build's `pnpm --filter @mlabs/web build`
  step, and `pnpm dev`.
- **Status:** ✓ fixed (commit `a5b882d`) — `apps/web/next.config.ts` renamed to `apps/web/next.config.mjs`; `pnpm build` re-verified clean; `pnpm dev` boots in <1s; S3 CORS preflight tests pass.

### Issue 2: `.jsonl` extension missing from rename.ts `TEXT_EXTENSIONS`

- **Severity:** **medium**
- **Repro:**
  1. Copy repo to tmp.
  2. `pnpm rename --namespace @example --slug example --display-name "Example App" --deeplink-host example.com`
  3. `grep -c "@mlabs/" .mstack/learnings.jsonl` → 10 (should be 0)
- **Expected:** Code-path references inside the JSONL `text` fields get
  rebranded (e.g. `@mlabs/db/schema` → `@example/db/schema`), matching what
  `.mstack/reviews/*.md` files do.
- **Actual:** `.mstack/learnings.jsonl` is silently skipped because
  `path.extname("learnings.jsonl") === ".jsonl"` and `.jsonl` is not in
  `TEXT_EXTENSIONS`. The shouldRewrite() short-circuits and the file passes
  through untouched.
- **Console errors:** none
- **Suspected cause:**
  `scripts/rename.ts:218–232` — `TEXT_EXTENSIONS` lists `.ts/.tsx/.js/.jsx/
  .mjs/.cjs/.json/.jsonc/.md/.mdx/.yaml/.yml/.css` but not `.jsonl`. The
  `.mstack/learnings.jsonl` is the only `.jsonl` consumer in this repo and
  it carries compound memory across forks, so getting stale `@<source-scope>/`
  refs is mildly confusing for the new fork's contributors.
- **Fix plan:**
  Add `".jsonl"` to `TEXT_EXTENSIONS` in `scripts/rename.ts`. One-line
  change. Verify by re-running the smoke in a fresh tmp copy.
- **Blocks:** Nothing in the current sprint; only affects fork-time hygiene.
- **Status:** ✓ fixed (commit `ef17fcf`) — `.jsonl` added to TEXT_EXTENSIONS; re-smoked via tmp copy: 10x `@mlabs/` → 10x `@example/` in learnings.jsonl, jq parses every line, 1 residual `@mlabs/<pkg>` is a literal placeholder pattern (intentional, regex correctly skips).
