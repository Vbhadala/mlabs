# Generated artifacts

What gets generated vs hand-written. Knowing the difference prevents
edits to generated files (which get overwritten) and untracked changes
to hand-written ones (which silently rot).

## Generated — don't edit; regenerate

| Artifact | Generated by | Committed? | Regenerate when |
|----------|--------------|------------|-----------------|
| `packages/db/drizzle/migrations/*.sql` | `pnpm db:generate` | YES | Schema change in `packages/db/src/schema/*` — the `check-migrations` lefthook hook will fail the commit if the SQL is stale |
| `apps/mobile/tailwind.config.js` | `pnpm gen:mobile-tw` | YES | `packages/config/src/design.ts` change — the `check-mobile-tailwind` lefthook hook will fail the commit if the config is stale |
| `apps/mobile/ios/`, `apps/mobile/android/` | `expo prebuild` | NO (gitignored) | Never under the managed workflow — only generate locally if you need to inspect native bridges |
| `pnpm-lock.yaml` | `pnpm install` | YES | Any `package.json` change in any workspace; pnpm regenerates on install |
| Better Auth tables (`user`, `session`, `account`, `verification`) | Better Auth runtime (lazy DDL on first auth) | N/A — lives in the database, not the repo | Auto. If you need to inspect, run `pnpm db:studio` |
| `apps/mobile/nativewind-env.d.ts` | NativeWind on first `expo start` | YES (TS needs the types) | Never manually; let NativeWind regenerate if it changes |
| `apps/web/.next/`, `apps/mobile/dist/` | `pnpm build` / Metro | NO (gitignored) | Per build |
| Turbo cache (`.turbo/`) | Turbo task runs | NO (gitignored) | Auto |

## Hand-maintained — review carefully

| Artifact | Owner | Note |
|----------|-------|------|
| `.npmrc` | hand-maintained | Hoist exclusions are load-bearing for Metro + pnpm — see forking-guide Tier 3 |
| `.syncpackrc.json` | hand-maintained | Carves out deliberate version divergences (mobile Tailwind 3 for NativeWind, etc.) — see forking-guide Tier 2 |
| `.github/renovate.json` | hand-maintained | Groups by ecosystem; React majors disabled across the board; tracked manually |
| `turbo.json` | hand-maintained | Task pipeline + `cache: false` on `db:*` tasks |
| `pnpm-workspace.yaml` | hand-maintained | Workspace globs |
| `package.json#pnpm.onlyBuiltDependencies` | hand-maintained | Allowlist for postinstall scripts — pnpm 10 default-denies; add deps here when you trust their install scripts |
| `apps/mobile/metro.config.js` | hand-maintained | Monorepo resolver config — required for Metro to find workspace packages |
| `public/.well-known/apple-app-site-association`, `assetlinks.json` | hand-maintained per fork | Per-client substitution; see HANDOVER.md.template §2 |

## When to regenerate

Quick checklist after pulling main or changing something:

- Touched any file in `packages/db/src/schema/`? → `pnpm db:generate`
- Touched `packages/config/src/design.ts`? → `pnpm gen:mobile-tw`
- Bumped a dep version in any `package.json`? → `pnpm install` (regenerates `pnpm-lock.yaml`)
- Changed `apps/mobile/app.config.ts` or added a native plugin? → only run `expo prebuild` if you're developing locally with the bare workflow; managed workflow regenerates on build
- Touched `packages/services/*` or `packages/api/*`? → no codegen needed; just `pnpm typecheck` to confirm the types still flow

## What's NOT generated (despite looking like it could be)

These files look like configuration that *might* be derivable from
something else, but they aren't. Treat them as hand-written:

- The mobile `babel.config.js` — explicit list of plugins/presets,
  partially driven by Expo SDK
- `apps/web/postcss.config.mjs` — Tailwind 4 needs a real PostCSS config
- `apps/web/next.config.ts` — Next.js config, including `transpilePackages`
  for workspace packages
- Any `tsconfig.json` in `apps/*` — extends `@mlabs/tsconfig` but apps
  also add their own `paths` and `include` patches
- `apps/mobile/app.config.ts` — Expo app metadata; substitute via the
  Phase 10 rename script when it lands, otherwise per-fork

## Pre-commit gates (lefthook)

The hooks at `.lefthook/pre-commit/*` enforce regeneration discipline:

- `check-migrations` — fails the commit if `packages/db/src/schema/`
  changed but `packages/db/drizzle/migrations/` doesn't have a paired
  SQL file
- `check-contrast` — fails if `packages/config/src/design.ts` produces
  any token pair below WCAG AA contrast thresholds
- `check-mobile-tailwind` — fails if `apps/mobile/tailwind.config.js`
  is out of sync with `packages/config/src/design.ts`

If any of these fire on a generated-file edit you forgot to regenerate,
that's the system doing its job. Run the matching `pnpm` command and
re-stage.
